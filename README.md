# 美容院ホームページ（PrivateSalon）

- 個人経営の美容院を想定し、大人の女性向けの落ち着いた雰囲気と、清潔感を重視したWebサイトです。
  WordPressで構築後、セキュリティとコスト面を考慮し、静的サイトとしてGitHub Pagesにデプロイしています。

## 制作の背景・目的

- Web制作ポートフォリオ用として作成しました。個人経営者の方でも日々の更新（画像の差し替えなど）が管理画面から直観的に行えるようWordPressを採用しました。
  あわせて、サーバー維持費（ランニングコスト）を最小限に抑えつつ、セキュリティリスクを低減させる構成を実務的な視点で提案・実装しました。

## 技術スタック

- **CMS**: WordPress
- **静的化**: Simply Static
- **ホスティング**: GitHub Pages
- **お問い合わせ**: Formspree（外部API連携）
- **その他**: HTML5 / CSS3 / JavaScript

## 主な機能

1. **予約デモフォーム:** 送信後のフィードバックメッセージ表示機能をJSで実装
2. **お問い合わせフォーム:** Formspreeを活用したメール通知機能
3. **店舗情報:** Google Map API / iframeによる地図埋め込み

## 工夫した点

- **ハイブリッドな運用構成**:</br>
  WordPressの管理画面による効率的な制作ワークフローと、GitHub Pagesによる静的配信を組み合わせました。
    - コスト面: クライアント側のランニングコストを0円に抑制。
    - 静的サイト化により、WordPress本体への外部攻撃リスクを完全に遮断。
    - 保守運用: 専門知識が必要な公開作業（静的化・デプロイ）はエンジニアが担当することで、サイトの品質と安全性を担保する運用モデルを想定しています。
- **外部サービスの活用**:</br>
  静的サイトではPHPが動作しないため、フォーム機能には Formspree を採用。実用的ビジネスサイトとしての導線を設計しました。
- **トラブルシューティング**:</br>
  GitHub Pagesへのデプロイ時にCSSが反映されない問題が発生した際、GitHub Pagesの内部エンジン（Jekyll）がアンダースコア（\_）で始まるディレクトリを無視する仕様が原因であると特定。.nojekyllファイルを配置することでこの挙動を回避し、表示を正常化させました。
- **ユーザー体験とデータの整合性**:</br>
  予約フォームはデモ用ですが、「送信したつもりなのに反応がない」というユーザーの不安を解消するため、JavaScriptを用いて即座に完了メッセージを表示させる制御を行いました。事務職で培った「正確な情報の伝達」と「相手の立場に立った配慮」を、コードの挙動にも反映させています。

## 予約・お問い合わせ機能の設計意図

- 予約フォーム（デモ実装）:
  静的環境ではPHPによるメール送信やDB保存ができないため、JavaScriptによるフロントエンド制御のみを実装しています。
    - 送信ボタン押下時に、ユーザーへ「送信完了（デモ用メッセージ）」を即座にフィードバックする仕組みを構築し、ユーザー体験（UX）を損なわない配慮を行いました。
- 実運用への提案:
  実際の店舗運用においては、以下のいずれかの手法を提案することを想定しています。
    1. 外部予約システムの統合: ホットペッパービューティーや専用予約サービスへのリンク・埋め込み（セキュリティと管理コストの最適化）。
    2. API連携: バックエンド機能のみを外部API（Formspreeの有料プランやGoogle Forms API等）と連携させ、静的サイトのままデータを受け取る。

## 制作を終えて（技術選定の考察）

- 本プロジェクトでは、ポートフォリオとしての公開コストを抑えるために「WordPressの静的化（GitHub Pages）」を採用しました。しかし、制作過程を通じて、実務における本構成の課題も再認識しました。
    - 運用の現実性:</br>
      静的化を行うたびに不要なコードの削除やパスの修正といったエンジニアによる手作業が発生するため、更新頻度が高いサイトでは人的コストが膨らむというデメリットがあります。
    - クライアントの利便性:</br>
      HTMLの知識がないクライアント様が、管理画面から自らお知らせや画像を更新するといった「セルフ更新」が困難になります。
    - 結論:</br>
      実務においては、効率的な運用とトータルコストの最適化を優先し、レンタルサーバーを利用した動的運用（WordPressの標準運用）を第一に提案すべきであると学びました。

    今回の挑戦で、静的・動的それぞれのメリットとトレードオフを深く理解することができました。今後は、技術的な興味だけでなく、「クライアントにとっての真の利益（運用効率とコストのバランス）」を常に考え、状況に応じた柔軟な提案ができるエンジニアを目指します。
